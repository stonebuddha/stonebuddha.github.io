Implementing an interactive theorem prover for First Order Logic

2016-12-07 | Theorem Proving

ML-family languages, including OCaml and Haskell, are mainstream functional programming languages nowadays. You may know ML abbreviates Meta Language, but what the heck is meta?

ML is originally proposed to implement trustworthy theorem provers. It is difficult to make a theorem prover fully trustworthy because programmers always make mistakes. The goal of ML is to devise some mechanism, which could reduce the part we must trust in a theorem prover. How does ML achieve this? Let’s think about how a theorem is proved. First of all we need a logic framework to state theorems as well as proofs. In this post I take First Order Logic (FOL). Then we should figure out what basic axioms and inference rules are in this framework. Because there are several different frameworks for FOL, I just pick one randomly. Finally we get to prove a theorem, which may be in the following form:
$$hyp_1,hyp_2,\cdots,hyp_n \vdash concl$$
where $hyp_i$ is the $i$-th hypothesis and $concl$ is the conclusion. Hypotheses and conclusions are formulas. We construct a proof tree whose root is the theorem to prove, leaves are axioms, and edges are instantiations of inference rules.

The proof trees turn out to be traditional data structures. To mimic the proof process, we can define a data type theorem to represent proved theorems. Each constructor of theorem is either an axiom (with no children) or an inference rule (with several children). Moreover, we must carefully control the use of inference rules to make them sound. This means that if the implementation of axioms and inference rules (which is typically small) is correct, every theorem which could be produced in our implementation must be sound! We can even discard the storage proof trees to get memory efficiency.

SML and OCaml also provides mechanisms to write annotations and anti-annotations, which allow users to manipulate proof objects using host languages. From the user’s prospective, ML shows up as a script language to prove theorems.

You may notice that the proof in our current system is constructed backwardly. It is opposite to how human beings think about proving theorems. We’d like to start from the goal, refine it step by step, till every subgoals are just axioms. This kind of refinement-based proof is forward. Forward approach provides one more feature, that we can implement tactics to help users prove much more easily! A tactic is a summary of proof scripts to prove some common goals.

How to extend our system with the ability to do forward proof, without sacrifice of soundness? I’d call that continuation-style-proof-construction.
